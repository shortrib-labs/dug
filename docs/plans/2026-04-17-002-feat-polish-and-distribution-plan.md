# Phase 3: Polish & Distribution

## Overview

Production-ready packaging for dug: custom help text, man page, Homebrew formula, README, shell completions, golden-file tests, and performance validation. The goal is making dug installable and discoverable ŌĆö a tool people can `brew install` and immediately use as a dig replacement that shows what the system resolver actually does.

This phase has no architectural changes. All code changes are in the CLI layer (help text, version output) and build/packaging infrastructure.

## Current State

- `--version` works via ArgumentParser (`0.1.0`)
- `--help` is auto-generated by ArgumentParser ŌĆö shows `USAGE: dug [<raw-args> ...]` which is useless for dig-style flags
- Shell completions work via `--generate-completion-script zsh` but only complete `--help`/`--version` (not `+` flags)
- No man page, no README, no Homebrew formula
- No golden-file regression tests
- No performance benchmarks

## Implementation Sequence

### Step 1: Custom --help Text

ArgumentParser's auto-generated help doesn't know about `+` flags, `@server`, or dig-style type/class arguments. Replace with custom help output that matches dig's help style.

**Approach:** Override `helpMessage()` on `Dug` or intercept `--help`/`-h` in the raw args before ArgumentParser sees them. Since we use `.allUnrecognized`, ArgumentParser passes `--help` to its own handler ŌĆö we'd need to check for `-h` in `rawArgs` before parsing.

**Actually simpler:** Use ArgumentParser's `abstract` and `discussion` fields in `CommandConfiguration` for a brief overview, then add a custom `HelpCommand` or detect `--help` in `run()` and print comprehensive usage.

**Content to cover:**
- Synopsis: `dug [@server] [+flags] [-flags] [name] [type] [class]`
- Server specification: `@8.8.8.8`, `@2001:4860:4860::8888`
- Record types: A, AAAA, MX, NS, SOA, CNAME, TXT, SRV, PTR, CAA, ANY
- Output flags: `+short`, `+traditional`, `+noall +answer`, `+cmd`/`+nocmd`
- Behavioral flags: `+tcp`, `+dnssec`, `+norecurse`, `+time=N`, `+tries=N`
- Transport: `+validate` (dug-specific DNSSEC validation probe)
- Debug: `+why` (shows resolver selection reason)
- Reverse lookup: `-x ADDRESS`
- Dig compatibility note: "Most dig flags work. dug defaults to the macOS system resolver."

### Step 2: Man Page (`dug.1`)

**Format:** roff/mandoc (standard man page format). Install to `share/man/man1/dug.1` via Homebrew.

**Sections:**
- NAME
- SYNOPSIS
- DESCRIPTION
- OPTIONS (flags grouped by category)
- OUTPUT FORMAT (enhanced vs traditional vs short)
- RESOLVER MODES (system vs direct, fallback triggers)
- DNSSEC (system validation vs direct RRSIG)
- EXAMPLES (10-15 practical examples)
- COMPATIBILITY (differences from dig)
- KNOWN ISSUES (macOS 26 regression, mDNSResponder DNSSEC limitations)
- SEE ALSO (dig, dscacheutil, scutil)

**Alternative:** ArgumentParser has a `generate-manual` plugin that produces man pages from command metadata. Given our `.allUnrecognized` parsing, this won't capture dig-style flags. Write the man page manually.

### Step 4: Shell Completions

ArgumentParser generates basic completions via `--generate-completion-script`. These only complete `--help`/`--version` since all other args are `.allUnrecognized`.

**Decision from origin plan (see origin):** "Ship ArgumentParser's auto-generated completions only (custom `+flag` completions are maintenance burden for negligible value)."

Follow this decision. Install the generated scripts via Homebrew formula. Support zsh (macOS default), bash, and fish.

### Step 5: Homebrew Formula

**Separate repo:** `shortrib-labs/homebrew-tap`

**Formula contents:**
```ruby
class Dug < Formula
  desc "macOS-native DNS lookup utility using the system resolver"
  homepage "https://github.com/shortrib-labs/dug"
  url "https://github.com/shortrib-labs/dug/archive/refs/tags/v0.2.0.tar.gz"
  sha256 "..."
  license "MIT"

  depends_on :macos
  depends_on xcode: ["15.0", :build]

  def install
    system "swift", "build", "-c", "release", "--disable-sandbox"
    bin.install ".build/release/dug"
    man1.install "dug.1"

    # Shell completions
    output = Utils.safe_popen_read(bin/"dug", "--generate-completion-script", "zsh")
    (zsh_completion/"_dug").write output
    output = Utils.safe_popen_read(bin/"dug", "--generate-completion-script", "bash")
    (bash_completion/"dug").write output
    output = Utils.safe_popen_read(bin/"dug", "--generate-completion-script", "fish")
    (fish_completion/"dug.fish").write output
  end

  test do
    assert_match version.to_s, shell_output("#{bin}/dug --version")
    assert_match "example.com", shell_output("#{bin}/dug +short example.com")
  end
end
```

**Installation:** `brew tap shortrib-labs/tap && brew install dug`

**Versioning:** Tag `v0.2.0` for the Phase 2 release (direct DNS fallback). Phase 1 was `v0.1.0`.

### Step 6: Golden-File Tests

Capture `dig +short` output for common queries, compare against `dug +short`. These are regression tests, not unit tests ŌĆö they validate end-to-end output format.

**Approach:** Process-level tests that spawn the `dug` binary and assert on stdout. Use `Process` in Swift Testing or shell scripts in the Makefile.

**Test cases:**
- `dug +short example.com` vs `dig +short example.com` ŌĆö A records match
- `dug +short example.com MX` vs `dig +short example.com MX` ŌĆö MX format matches
- `dug +short @8.8.8.8 example.com` vs `dig +short @8.8.8.8 example.com` ŌĆö direct DNS matches
- `dug -x 8.8.8.8` vs `dig -x 8.8.8.8` ŌĆö reverse lookup format matches
- `dug example.com` output structure ŌĆö sections present in correct order

**Note:** Golden files can't be exact matches due to TTL variation and timestamp differences. Compare structure and record data, not full output.

### Step 7: Performance Validation

**Target from origin plan:** <10ms overhead.

**Measurement:**
```bash
DYLD_PRINT_STATISTICS=1 .build/release/dug +short example.com 2>&1 | grep "Total pre-main"
```

Also measure with `time` for wall-clock comparison against dig:
```bash
time dug +short example.com
time dig +short example.com
```

Document results in README's "Performance" section.

### Step 8: Hardened Runtime

For notarized distribution (outside Homebrew), enable hardened runtime:
- Add `--enable-hardened-runtime` to the release build
- No special entitlements needed (dug doesn't use restricted APIs)
- Sign with Developer ID for notarization

This is optional for Homebrew (Homebrew handles signing) but needed for direct `.pkg` or `.dmg` distribution.
